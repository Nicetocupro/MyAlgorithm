## 三种写法对比

你的写法本质上是对的，但**用两个 if-else 分支重复处理了本可以对称统一的逻辑**。对比后能看出几个层次的问题。

### 核心问题：你把"求交集"和"移动指针"耦合成了两条并行分支

你写的是：
- 情况 A：`begin1 <= begin2 && end1 >= begin2`（区间1包住区间2的起点）
- 情况 B：`begin2 <= begin1 && end2 >= begin1`（区间2包住区间1的起点）
- 情况 C：无交集

但其实 A 和 B 是**同一件事的两种对称写法**——你在用条件分支模拟本该由 `max/min` 自动完成的事情。这也是为什么你的代码行数是别人的 3 倍以上。

### 别人的写法好在哪

**方法一**（最简洁）：
```cpp
int start = max(firstList[i][0], secondList[j][0]);
int end = min(firstList[i][1], secondList[j][1]);
if (start <= end) ans.push_back({start, end});
```
这一行直接把你的两个 if 分支和"求交集区间"这件事统一了：**交集的起点永远是两个起点的较大值，终点永远是两个终点的较小值**，`start <= end` 就自动判断了是否有重叠，不需要你手写三种大小关系比较。这是区间问题里最值得记住的技巧（也是 56/57/1288 等题的通用范式）。

指针移动的判断也更干净：`end1 < end2` 就推进 `i`，否则推进 `j`——因为**结束更早的区间不可能再与后续区间产生交集了**，可以放心丢弃。你的代码里这个逻辑被拆散在两个分支内部，正确性一样但可读性差很多。

**方法二**：思路和方法一完全一样，只是把"无交集"和"有交集"显式分成两个 if 块，可读性稍微啰嗦一点，但在面试时如果你觉得默写 `max/min` 一行式没把握，这种显式写法反而更不容易出错，属于"更啰嗦但更保险"的版本。

### 你的代码可以怎么优化

把你的整个 while 循环体替换为：
```cpp
while (i < firstList.size() && j < secondList.size()) {
    int lo = max(firstList[i][0], secondList[j][0]);
    int hi = min(firstList[i][1], secondList[j][1]);
    if (lo <= hi) result.push_back({lo, hi});
    if (firstList[i][1] < secondList[j][1]) i++;
    else j++;
}
```
行数从 ~35 行压缩到 ~6 行，逻辑完全等价。

### 面试进步点

1. **看到"两个有序区间求交/求并"，先问自己：交集的边界能不能用 max/min 直接表达？** 很多人第一反应是分情况讨论（像你这样），但区间问题绝大多数可以用 `max(start)/min(end)` 统一表达，减少分支就是减少出错点。
2. **指针移动的判断只需要关心"谁先结束"**，不需要在每个分支里重复判断——把"求交集"和"移动指针"这两件事在代码结构上拆开，会让整体逻辑更清晰，是这道题现场写代码时最容易体现工程素养的地方。
3. 如果面试官追问扩展（比如你之前提到的 1000+ 传感器场景），可以主动提到：两两 pairwise 求交集是 O(n×m)，如果要多路（k 个区间列表）求交，更好的做法是**扫描线 + 计数**（差分数组），这是从这道题到工程场景的自然延伸，说出来会加分。
